# 要求仕様書

## 1. 本書の位置づけ

本書は、Markdown Viewer（以下「本アプリ」）について「何を実現するか」をまとめた要求仕様書である。具体的な実現方法・技術構成については [spec.md](spec.md)（ソフトウェア仕様書）を参照する。

## 2. 背景・目的

- 既存の `app/md-vault`（クラウド版。要認証、フロントエンド＋バックエンド構成）、`app/md-vault_webr`（ローカル専用版。localhost限定・認証なしで自宅PC内のみで完結）は、いずれも「不特定多数への一般公開」を前提としていない。特に`md-vault_webr`は要求仕様書N-01/N-02で「フォルダの実データは外部に送信しない」「localhost以外からのアクセスは受け付けない（認証なし）」と明記されており、そのまま外部公開すると誰でも任意のフォルダを読み書きできてしまう
- 本アプリは、上記2アプリとは異なる目的で作る「閲覧・編集専用の公開Webアプリ」である。対象フォルダは利用者自身のローカル環境から都度選択してもらう方式とし、サーバー側は一切フォルダの中身（編集内容を含む）に触れない
- ブラウザのFile System Access API（`showDirectoryPicker()`）を使い、選択したフォルダの内容をブラウザ内だけで読み書きしてMarkdownを表示・編集する。ファイル内容はサーバーに送信されないため、バックエンドを持たない静的サイトとして誰でもアクセスできる場所に公開できる
- 既存2アプリとは併存させる。「ちょっと人に見せる・どの端末からでも閲覧だけしたい」場面では本アプリ、自宅での編集作業には`md-vault_webr`、外出先からの編集には`md-vault`、と使い分ける
- 対象フォルダはObsidianのVault（保管庫）のような、Markdownファイルを格納した一般的なフォルダを想定するが、アプリ自体はObsidian固有の機能・用語には依存しない

## 3. 用語定義

| 用語 | 説明 |
| --- | --- |
| クラウド版 | 既存の`app/md-vault`。要認証、フロントエンド＋バックエンド構成 |
| ローカル専用版 | 既存の`app/md-vault_webr`。localhost限定・認証なしで自宅PC内のみ完結 |
| File System Access API | ブラウザ上でユーザーが明示的に許可したローカルフォルダ・ファイルに直接アクセスできるWeb標準API。Chrome/Edge等Chromium系ブラウザのみ対応 |

## 4. 想定利用者・利用シーン

- 利用者: 本アプリのURLにアクセスした任意の人（不特定多数）。ただし各利用者は自分のPC上のフォルダしか閲覧できない（他人のフォルダにはアクセスできない）
- 想定される利用シーン:
  - 自分のPC上のMarkdownフォルダを、インストール作業なしにブラウザだけでさっと閲覧したい
  - 会社のPCや外出先の別端末など、ふだん使わない環境でも同じフォルダを一時的に閲覧したい
  - 用途ごとに分かれた複数のフォルダを切り替えて使いたい
  - 目的のノートをファイル名や本文から素早く探したい

## 5. 機能要求

| No | 要求 |
| --- | --- |
| F-01 | 利用者が自分のPC上の任意のフォルダを選択できること |
| F-02 | 選択したフォルダ内のMarkdownファイルをフォルダ階層のツリーで一覧できること |
| F-03 | 選択したファイルの内容を閲覧・編集できること（frontmatterはプレビューから除くが、編集領域では生テキストのまま扱う） |
| F-04 | Markdown内の相対パスで参照された画像を解決して表示できること |
| F-05 | 一度選択したフォルダへのアクセス権限（読み書き）を次回訪問時も再利用できること（フォルダ選択のやり直しを毎回求めない） |
| F-06 | File System Access APIに対応していないブラウザでは、その旨を案内すること |
| F-07 | ファイル名・本文を対象に検索でき、本文検索では該当箇所のスニペットを表示できること |
| F-08 | ダークモードを手動で切り替えられ、設定が次回以降も保持されること |
| F-09 | サイドバーの幅を調整でき、非表示/再表示を切り替えられること。設定は次回以降も保持されること |
| F-10 | 複数のフォルダを選択履歴として保持し、画面上から切り替え・追加・削除できること。履歴は最大3件までとし、超えた分は最も長く開かれていないものから自動的に削除する。また30日間開かれなかったフォルダも履歴から自動的に削除する（実フォルダには影響しない） |
| F-11 | 編集した内容を保存でき、保存後もObsidian等の他アプリで正しく開けること（frontmatter・フォルダ構造を破壊しないこと） |
| F-12 | 表示モードを「編集のみ」「両方」「プレビューのみ」から選べ、設定が次回以降も保持されること |
| F-13 | 「両方」表示時、編集とプレビューの左右を入れ替えられること |
| F-14 | 「両方」表示時、編集⇄プレビューの相対スクロール位置を同期するON/OFFを切り替えられること |

> 既存2アプリにあった「画像アップロード」「ログイン機能」は、本アプリのスコープ外とする（8章参照）。

## 6. 非機能要求

| No | 要求 |
| --- | --- |
| N-01 | フォルダの実データ（ファイル内容、編集内容を含む）は、いかなる場合もサーバーに送信・保存しないこと。読み取り・書き込みともにブラウザ内で完結すること |
| N-02 | 不特定多数からのアクセスを受け付けるが、各利用者がアクセスできる（読み書きできる）のはその利用者自身が明示的に選択・許可したローカルフォルダのみであるため、認証機能は設けない。書き込み権限（`readwrite`）も同様に、利用者本人がブラウザの許可ダイアログで明示的に許可したフォルダに限られる |
| N-03 | 無料で利用できるツール・サービスを優先して選定すること |
| N-04 | 既製品SaaSの利用ではなく、開発者本人が仕組みを理解しながら構築できる自前実装であること |

## 7. 制約条件

- File System Access APIはChrome・Edge等Chromium系ブラウザのみ対応。Firefox・Safari（iPhoneのSafariを含む）では利用できない
- バックエンドを持たない静的サイトとして構成する（サーバーサイドの処理を必要としない）

## 8. スコープ外（本アプリで対応しないこと）

- 画像のアップロード機能
- ログイン等の認証機能
- File System Access API非対応ブラウザ向けの代替実装
- Obsidianの`[[wikilink]]`のようなノート間リンクの解決・グラフ表示
- 自動テスト（CI等）の整備

これらは将来的に必要になった場合、改めて要求を整理したうえで追加を検討する。
